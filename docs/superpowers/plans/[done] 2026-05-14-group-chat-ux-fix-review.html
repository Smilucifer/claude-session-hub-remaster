Bug 修复
FIX 步骤 1: Array.find 回调参数错位
文件: GroupChatLayout.svelte:61 | 改动量: 1 行
Array.find 的回调签名是 (element, index, array)，当前代码 .find((_pid, _label, runId) => runId === ev.run_id) 中，第三个参数 runId 实际是整个数组，与字符串比较永远返回 false。
bus event listener 完全失效 — thinking 状态永远不显示，permission prompt 永远不弹出。这是 Group Chat 中两个核心交互功能的静默失效。
// 修复前（broken） const pid = participantRunIds().find((_pid, _label, runId) => runId === ev.run_id); // 修复后 const pid = participantRunIds().find((p) => p.runId === ev.run_id);
影响: thinking 折叠面板恢复显示 影响: permission prompt 恢复弹出 风险: 极低（单行修复）
FIX 步骤 4: 参与者名字显示为 UUID
文件: GroupChatLayout.svelte:577 | 改动量: ~10 行
getParticipantInfo() 按 participant.id 查找，当返回 undefined 时 fallback 到原始 UUID（如 28bae63d-f702-49ee-ba8d-8320400074a5），用户看到的是无意义的 ID 而非角色名。
在 primary lookup 失败时，增加 run_id 二次反查（用 resp.run_id 匹配 p.participant.run_id），最终 fallback 到 "Unknown" 而非 UUID。
影响: 参与者名字正确显示为角色名 风险: 低（防御性 fallback）
FIX 步骤 5: Idle 状态颜色混淆
文件: GroupChatLayout.svelte:356-388 | 改动量: ~5 行
Idle（就绪等待输入）和 undefined（未加载）共用灰色圆点，用户无法区分参与者是"还在启动"还是"已经就绪"。
拆分 statusColor 的 default case：Idle 使用 bg-blue-400/70 + label "Ready"，与 undefined 的灰色区分。
影响: 三种状态（Starting/Ready/Running）视觉可区分 风险: 极低（纯 CSS）
体验优化
ENHANCE 步骤 2: 消息发送后 UI 冻结
文件: GroupChatLayout.svelte:314-335 | 改动量: ~15 行
handleSend() 用 await api.sendGroupChatMessage() 阻塞等待整个 turn 完成（可能 30s+），用户在这期间看到空屏幕，体验极差。
改为 fire-and-forget + 组件内局部 polling：先启动 setInterval 每 1.5s 调用 api.getGroupChat() 刷新 detail，同时 await sendPromise。polling 在 await 之前启动，确保 turn 执行期间能看到增量 JSONL 快照。
方案设计时曾考虑复用 groupChatStore.sendMessage()，但 4 路 review 一致确认不可行：组件完全不引用 store，两者维护独立的状态副本。改为组件内 self-contained polling 是最小改动路径。
// 修复后 handleSend 核心逻辑 const sendPromise = api.sendGroupChatMessage(roomId, messageToSend); composerText = ""; // Polling 必须在 await 之前启动 const pollTimer = setInterval(async () => { try { const updated = await api.getGroupChat(roomId); if (detail?.id === roomId) detail = updated; } catch { /* ignore */ } }, 1500); try { const updated = await sendPromise; if (detail?.id === roomId) detail = updated; } finally { clearInterval(pollTimer); }
影响: 发送后 UI 不再冻结 影响: participant 状态随 polling 实时更新 风险: 低（自包含，不引入新依赖）
后续增强（暂缓）
DEFER 步骤 3: MessageDelta 实时文本流
文件: GroupChatLayout.svelte:57-88 | 改动量: ~25 行（被低估）
bus event 中的 message_delta 包含参与者正在生成的文本片段，但组件只从 resp.preview（turn 完成后的最终快照）渲染，无法看到实时流式文本。
新增 messageTexts: Map<runId, string> 状态累积 delta text，在 timeline 模板中优先显示流式文本，在 message_complete 时用完成事件替换。但步骤 2 的 polling 已覆盖增量更新，此步骤是锦上添花。
状态: 4/4 provider 建议暂缓 原因: polling 已覆盖，边际收益有限 前提: 需先确认 pipe_exec 路径是否 emit MessageDelta
不做（4 路 review 一致否决）
SKIP Clowder-ai 架构迁移
Socket.IO / Zustand / BubbleReducer / Optimistic Rendering / 序列号 gap detection
提案将 ClawGO Group Chat 的问题归因于"通信层不够强大"，建议迁移到 Clowder-ai 的 Socket.IO + Zustand + BubbleReducer 架构。
4/4 provider 一致否决：ClawGO 的 BusEvent（20+ 变体）、BroadcastEmitter（双通道 + seq）、orchestrator（incremental JSONL）已经构成完整基础设施。问题不是基础设施不存在，而是组件层绕过了它。Socket.IO 在 Tauri 本地桌面应用中是过度工程化。
